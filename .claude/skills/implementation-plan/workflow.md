# Implementation Plan Workflow

## 1. 入力の読み込み

- P2 の確定版 AC/QC（Jira コメント）
- 設計書・ADR（あれば）。bugfix / chore など P3・P4 を実施しないルートでは P2 の出力から直接計画する
- P2.5 を実施した場合は `IMPLEMENTATION_PROMPT.md`（プロトタイプ合意の制約）

## 2. 計画の作成

`template.md` に従い、以下を含める。

- 実装順序とその理由（なぜこの順か）
- 各ステップの TDD / DIRECT 区分
  - TDD: テストで駆動できる作業。`tdd-cycle` で実装する
  - DIRECT: テストで駆動できない作業（環境構築・設定変更・スキーマ適用など）。実施手順と検証手順を必須にする
- 各ステップが満たす AC-ID・設計要素
- 変更対象と、触らない範囲（それぞれ理由つき）
- リスクと検証方針
- 人間確認ポイントの配置

## 3. 刻み方の検討

1 ステップは PR レビュー可能な粒度（1〜数コミット）に保つ。議論の中心は刻み方と確認ポイントの配置。計画が大きくなりすぎる場合はチケット分割を提案する。

## 4. 承認と記録

- 人間の承認を得る
- draft PR を作成し、説明文に計画を書く（branch 名は `feat/{チケット番号}-short-desc`、PR タイトルは `{チケット番号}: 要約`）

## 注意事項

- 実装方針は使い捨ての短期コンテキスト。長期価値のある判断理由は P7 の短期コンテキストの昇格で ADR・設計書へ移す
- 計画時点で新しい技術判断が見つかったら P3 へ戻る（Intent Delta を Jira へ追記してから）
- bugfix は「再現テストを書く」が最初のステップになる（簡易計画は PR 説明 1 段落でよい）
