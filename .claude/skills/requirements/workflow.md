# Requirements Workflow

## 1. 入力の読み込み

- GitHub Issue本文（Product Intent）と既存の Intent Delta コメントを読む
- タイプ（`type-*` ラベル）を確認し、タイプ別の焦点（`checklist.md`）を把握する
- Issue の要求を自分の言葉で言い直し、理解のずれがないか人間に確認する

## 2. 受入れ条件の具体化

Issue の受入れ条件を 1 件ずつ検査する。

- 曖昧: 読み手によって解釈が分かれる表現を指摘する
- 矛盾: 条件同士、または対象範囲との不整合を指摘する
- 漏れ: エッジケース（エラー、空、境界値、権限、並行操作）の不足を指摘する

不明点は推測で埋めず、人間に確認する。確認できるまでは 🟡 または 🔴 マーカーを付けて残す。

## 3. 品質条件の導出

`checklist.md` の標準質問リストを走査し、該当する観点だけを測定可能な基準として書き出す。該当しない観点は「他は対象外」の 1 行に畳む。全Issue に全観点の記述を並べない。

## 4. ID・マーカー・タグの付与

- 受入れ条件に `AC-1`…、品質条件に `QC-1`… の ID を振る。以後の実装計画・テスト・対応表・検証記録はこの ID で参照される
- 各項目に信頼性マーカーを付ける
  - 🔵 根拠あり（Issue・人間の回答・既存仕様に出典を明記）
  - 🟡 妥当な推定（根拠が間接的。人間の確認が必要）
  - 🔴 根拠なし・要確認
- 各 AC・守るべき振る舞いにテストレベル（unit / integration / E2E）を割り当てる。原則は「検証できる最も下位のレベルで検証し、下位で表現できないものだけ上位へ」。プロダクトの preset があればそれに従う
- 各 QC に検証手段タグ（CI / テスト / 手動）を付ける。CI・テストはゲート 1（P6 exit）、手動はゲート 2（P7）で検証される

## 5. 確認ループ

🟡・🔴 が残っている間は確定しない。人間に確認すべき点をまとめ、回答しやすい形で質問する。

- 一度に聞く項目を増やしすぎない
- 回答を得たら 🔵 に更新するか、対象外として畳む

## 6. 確定と記録

人間の承認を得たら、`output-format.md` の形式で GitHub Issueへコメントする（Issue本文は書き換えない。Intent Delta として追記する）。

## 注意事項

- プロダクト判断が必要な不明点は `product-intent` の確認ループへ差し戻す（PdM の回答が必要なもの）
- タイプにより焦点が変わる: bugfix は再現条件と回帰範囲、refactoring は不変条件と対象範囲、spike は問いの定義が中心
- 受入れ条件（ユーザー視点）と品質条件（技術視点）を混ぜない
