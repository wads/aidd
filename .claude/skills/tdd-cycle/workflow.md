# TDD Workflow

TDD はテスト技法ではなく、振る舞いを先に確定させる設計と検証の規律である。このプロセスでは、ゲート 1（P6 exit、テスト green）の信頼性が「テストが実装と独立に書かれたこと」に依存するため、test-first は省略できない。

## 0. TODO リスト

実装する AC を、検証すべき振る舞いの TODO リストに分解する。

- 1 行 1 振る舞い。各行に対応する AC-ID を付ける
- 設計書の「守るべき振る舞い」も行に含める
- リストから 1 つだけ選んでサイクルに入る。残りには手を付けない
- サイクル中に気づいた振る舞いはその場で実装せず、リストに追記する

## 1. Red: 失敗するテストを書く

- TODO リストから選んだ 1 つの振る舞いについて、テストを 1 つ書く
- テストの根拠は AC・設計書・チケットであり、実装コードではない
- **実行して、失敗を観察する。** 「落ちるはず」と仮定して進まない
- **正しい理由で落ちたことを出力で確認する**（コンパイルエラーや別件のエラーで落ちているなら、それは Red ではない）
- **予期せぬ Green は停止事象**: 新しいテストが一度も赤にならず通ったら即停止し、人間へ報告する。振る舞いが既に存在する / テストが何も検証していない / 仕様を誤解している、のいずれかである

## 2. Green: テストを通す

最小の変更でテストを通す。戦略は確信度で選ぶ。

- **明白な実装**（デフォルト）: 既知のパターンで確信があるなら、素直に実装する
- **仮実装 → 三角測量**（不確かなとき）: 根拠の薄い領域（未知のライブラリ、🟡 で確定した仕様、複雑なロジック）では、まずベタ書きで通し、2 つ目の例を追加して一般化を強制する。三角測量は AI の誤ったパターンマッチを暴く道具でもある
- 通すのに必要な分だけ書く。先回りの一般化・機能追加をしない

## 3. Refactor: 重複を除去する

- 主目的は重複の除去と意図の明確化。「なんとなく改善」はしない
- 振る舞いを変えない。テストが green のまま維持されることを実行して確認する
- リファクタ中に新しい振る舞いが必要だと気づいたら、TODO リストに戻す

## 4. 繰り返しと commit

- TODO リストの次の 1 行を選んで繰り返す
- commit はサイクルの切れ目で小さく刻む（人間が commit タイミングを判断する場合はそれに従う）

## 鉄の規律

1. 失敗するテストなしに本番コードを書かない
2. 失敗させるのに必要以上のテストを一度に書かない（テストの大量先行生成は後付けと同じ仕様ドリフトを生む）
3. 実装に合わせてテストを書かない（写経テストの禁止）。テストの出所は常に AC・設計書・チケット
4. テストを実行せずに結果を仮定しない

## ガイドライン

- 1 サイクルの変更は小さく保つ。根拠が薄い領域ほど歩幅を狭くする
- テストは振る舞いを検証する。内部実装の詳細に結合させない
- 外部依存のみモックする
- テストレベル（unit / integration / E2E）は P2 の割り当てに従う
- DIRECT 区分のステップ（環境構築・設定変更など）は TDD の対象外。実装計画の実施手順・検証手順に従う

## 受入れ条件との対応（出力契約）

- テスト名・ケースを受入れ条件（AC-ID）と対応付け、対応表を PR 説明に出力・更新する
- 設計書の「守るべき振る舞い」も AC と同様にテスト化する

## 停止条件

- 予期せぬ Green（上記）
- 計画からの逸脱（スコープ変更、想定外の問題、新しい技術判断）
- テストに過剰適合した不自然な実装しか思いつかないとき（仕様の誤解を疑い、人間へ確認する）

いずれも作り込まずに停止して人間へ確認する。要求や設計の矛盾を見つけた場合は、Intent Delta を Jira へ追記してから該当フェーズへ差し戻す。
