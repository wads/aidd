# Design Docs

## いつ使うか

- 責務の置き場所や例外ケースの扱いに設計判断が必要なとき
- 「守るべき振る舞い」を将来の変更者へ残す必要があるとき
- P7 の短期コンテキストの昇格で、設計判断を記録化するとき

書かない判断も正当である。設計書が必要かは「将来の変更者がこれを知らないと壊すか」で決める。

## 入力

- 確定版の受入れ条件・品質条件（P2 の Jira コメント、AC/QC ID）
- 関連する ADR（あれば）
- 既存の設計書・コード構造

## 出力

- 設計書 → `docs/design/{連番}-{チケット番号}-short-title.md`（`template.md` 参照）
- Jira チケットへ設計書のリンクをコメント

## 扱う Intent

- Design Intent（作成）/ Product Intent・Decision Intent（入力）

## exit 条件（ゲート）

- 受入れ条件の各 AC-ID について「設計のどこで満たされるか」を対応付けられる
- 🟡・🔴 マーカーが残っていない
- ADR と内容が重複していない（判断理由は ADR へ寄せ、参照する）
- 人間が設計を承認している

## 読むべき補助ファイル

- `workflow.md`
- `template.md`
