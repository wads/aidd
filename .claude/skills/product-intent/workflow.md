# Product Intent Workflow

この skill は、GitHub Issue作成そのものではなく、要求を AIDD で扱える Product Intent に整えることを主目的とする。

## 1. タイプの確定（必須）

最初にIssue のタイプを確定する。タイプの一覧・判定原則・テンプレートは `type-templates.md` を参照する。

- 依頼内容からタイプを提案し、人間の確認を得る。AI が推測で確定しない
- タイプにより本文テンプレート、GitHub Issue 課題タイプ、`type-*` ラベル、開発プロセスのルートが決まる
- feature 以外（bugfix / hotfix / refactoring / chore / spike）はエンジニア起点を想定した軽量テンプレートを使い、確認ループは必須項目の欠落時のみとする。以降のセクション 2〜5 は feature 向けの手順であり、他タイプは `type-templates.md` のテンプレートを埋めたらセクション 6 へ進む

## 2. 入力の受け取り

ユーザーから、メモ、口頭説明、既存Issue案、箇条書きのいずれかを受け取る。

入力が空に近い場合は、最初に以下をまとめて質問する。

1. 背景・意図
2. ユーザーストーリー
3. 作るもの
4. 対象範囲
5. 受入れ条件
6. 前提・制約
7. 関連情報

## 3. Issue案への構造化

受け取った内容を、以下の形式へ整理する。

この時点で不足している項目は推測で埋めない。不明な項目は `未確認` として残す。

```md
## 背景・意図
{なぜ作るのか、どんな課題や機会に対応するのか}

## ユーザーストーリー
{対象ユーザー} として、
{やりたいこと} したい。
なぜなら {得たい価値・解決したい課題} だから。

## 作るもの
{ユーザーに見える振る舞いとして何を作るのか}

## 対象範囲
- 対象画面・導線:
- 対象ユーザー・ケース:
- 対象外:

## 受入れ条件
- {条件または操作} のとき、{期待結果} になる
- {確認観点} を {確認方法} で確認できる

## 前提・制約
{なければ「特になし」}

## 関連情報
{なければ「特になし」}
```

## 4. 品質チェック

観点は `checklist.md` を参照する。

以下を確認する。

- 必要な内容がそろっているか
- 不備や矛盾がないか
- 過剰に実装方式へ踏み込んでいないか
- 読み手が迷う曖昧な表現がないか
- 受入れ条件が確認可能な粒度になっているか
- AI が推測で補っている箇所がないか

## 5. 確認ループ

品質チェックで不足、不備、過剰、不明瞭な点がある場合は、Issue を作成しない。

人間に確認すべき点をまとめ、回答しやすい形で質問する。

質問では以下を守る。

- 一度に聞く項目を増やしすぎない
- 必須の不足事項を優先する
- 何を判断するための質問かを明示する
- AI が推測で候補を確定しない
- ユーザーが回答したらIssue案を更新し、再度 `checklist.md` で確認する

このループは、Issue案が一定品質に達するまで繰り返す。

## 6. GitHub Issue 作成用の整理

- タイトルを生成
- `type-*` ラベルはタイプの対応表（`type-templates.md`）に従う
- 必要に応じて親 Issue を確認
- プロジェクト固有のラベル運用があれば、それに従って併せて付与する

## 7. 作成前確認

以下を提示して承認を求める。

- 作成先リポジトリ（Binding の `issue_repo`）
- タイプ（`type-*` ラベル）
- タイトル
- 親 Issue
- ラベル
- 説明

## 8. Issue 作成

承認後、Binding の `issue_repo` に対し、利用可能な GitHub Issue 作成ツール（例: `gh issue create`）で作成する。

パラメータ:

- 作成先: Binding の `issue_repo`（既定は対象 repo 自身）
- `labels`: `["type-{タイプ}"]`（プロジェクト固有ラベルがあれば追加）

## 9. 作成後

作成されたIssue 番号と URL を表示する。

## 注意事項

- 作成前に必ず確認を取る
- 担当者、スプリントは明示要求がない限り設定しない
- 不明点は推測で補わない
- 技術的な影響範囲、実装方式、詳細なテスト範囲の判断は必須責務にしない（タイプの確定は必須責務）
- 開発ルート（実施フェーズの構成）はタイプから自動で決まるため、Issue本文に書かない
- Issue本文は Product Intent を中心にする
