# タイプ別Issueテンプレート

タイプはIssue 作成の必須入力である。タイプごとに本文テンプレートと GitHub Issue 設定（課題タイプ・ラベル）が決まり、開発プロセスのルート（実施フェーズ）もこのタイプで決まる。

| タイプ | 内容 | ラベル / 課題タイプ |
|---|---|---|
| feature | 機能追加・仕様変更 | `type-feature` / Story |
| bugfix | 欠陥修正 | `type-bugfix` / Bug |
| hotfix | 緊急の欠陥修正 | `type-hotfix` / Bug |
| refactoring | 外部振る舞い不変の内部改善 | `type-refactoring` / Task |
| chore | 定型作業（依存更新、設定変更、軽微な文言） | `type-chore` / Task |
| spike | 調査・検証（コードは成果物にしない） | `type-spike` / Task |

タイプの判定原則:

- 受入れ条件の性質で分ける。「直っていること」なら bugfix、「変わっていないこと」なら refactoring、「ユーザーに見える新しい振る舞い」なら feature
- 複数タイプが混在する依頼は、Issue を分割するよう提案する
- 迷う場合は AI が推測で確定せず、候補と理由を提示して人間に確認する

## feature

`workflow.md` の Product Intent テンプレート（背景・意図、ユーザーストーリー、作るもの、対象範囲、受入れ条件、前提・制約、関連情報）をそのまま使う。確認ループも通常どおり実施する。

## bugfix

エンジニア起点を想定し、記入が数分で終わることを優先する。確認ループは必須項目の欠落時のみ。

```md
## 再現手順
1. {操作}

## 期待する挙動
{本来どうなるべきか}

## 実際の挙動
{何が起きているか。エラーメッセージ・ログがあれば貼る}

## 影響範囲
{誰が・どの画面/機能で影響を受けるか}

## 回帰の懸念箇所
{修正が影響しうる周辺機能。不明なら「未確認」}
```

## hotfix

bugfix と同じテンプレート + 緊急性の根拠。Issue 作成が事後になることを許容する（先に直し、Issue と記録を事後に整える）。

```md
## 緊急性の根拠
{影響の大きさ・広がり。なぜ通常の bugfix フローを待てないか}
```

## refactoring

```md
## 動機
{なぜ今やるか。放置した場合に何が悪化するか}

## 対象範囲
{触るモジュール・ディレクトリ}

## 不変条件（守るべき振る舞い）
{この作業で変わってはいけない外部振る舞い}

## 対象外
{今回は触らない範囲}
```

## chore

```md
## 作業内容
{何をするか}

## 完了の確認方法
{CI green、依存バージョン確認など機械的に確認できる方法}
```

## spike

```md
## 答えるべき問い
{この調査が終わったと言える問い}

## 背景
{なぜこの問いに答える必要があるか}

## time-box
{調査に使う時間の上限}

## 成果物の形
{ADR / 報告メモ。調査用コードはマージしない}
```
