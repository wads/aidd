# Context Snapshot Workflow

## 1. 情報の収集

Issue 番号を起点に以下を読む（GitHub Issue が索引。Issue のリンクとIssue 番号の grep で辿る）。

- GitHub Issue本文（Product Intent）と Intent Delta コメント（P2 確定条件、差し戻し記録、UI 合意）
- 関連 ADR・設計書（Binding の records_root 配下の `adr/`・`design/` を Issue 番号で grep）
- PR の実装方針・AC⇄テスト対応表・検証記録・必須申告セクション

## 2. 意図の復元

Product Intent に Intent Delta を古い順に適用し、「最新の意図」を復元する。原文の転載ではなく、適用後の姿を簡潔に書く。

## 3. HTML の生成

1 ファイルの HTML として生成する。セクション構成:

1. **Goal**: この開発単位で達成すること（1〜2 行）
2. **最新の意図**: Delta 適用後の受入れ条件・対象範囲（AC-ID つき）
3. **技術判断**: ADR の要約（採用・捨てた選択肢・トレードオフ）とリンク
4. **設計の要点**: 守るべき振る舞いを中心に（設計書へのリンク）
5. **検証状態**: AC-ID ⇄ テスト ⇄ 検証結果のマトリクス（未検証は明示）
6. **注意が必要な点**: 信頼性マーカーの残存サマリー（未解決の 🟡・🔴）、未検証項目、未確定事項
7. **昇格された記録**: 短期コンテキストから ADR・設計書へ移した内容（あれば）

ヘッダに生成日時と「これは生成ビューであり、最新の正は記録側にある」旨を明記する。

## 4. 添付と提示

- GitHub Issueへ添付する（ファイル名: `snapshot-{Issue 番号}-{YYYYMMDD}.html`）
- P7 で使う場合は、受入れ判断で人間が見るべき点（セクション 6）を先頭に要約して提示する

## 注意事項

- 記録に無い内容を推測で補わない。情報が見つからない項目は「記録なし」と書く
- 将来拡張: 入力範囲をプロジェクト全体（コード + 全 ADR・設計書）に広げたプロジェクトレベルのスナップショット。その場合の信頼性マーカーは、生成されるビュー自身の各記述に付与する（どの記述が記録に根拠を持ち、どれがコードからの推定か）
