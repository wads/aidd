# UI Prototyping Workflow

UI プロトタイピングの全体フローを管理する。

## フェーズ

1. 要求の構造化
2. variant / preset の確認
3. プロトタイプ生成
4. プレビュー案内
5. フィードバック反映
6. 実装ハンドオフ生成

## 基本フロー

```text
Phase A: 要求の受け取りと構造化
  -> 目的、デバイス、主要機能、状態を整理
  -> 曖昧点があれば確認

Phase B: variant / preset の確定
  -> project-config.md を読む
  -> stack に応じて読む variant を決める
  -> 既存コンポーネント、デザイントークン、出力先を確認

Phase C: プロトタイプ生成
  -> variant の生成ルールに従ってファイルを作る
  -> 主要状態のプレビュー用ファイルも作る

Phase D: フィードバック反映
  -> 修正要求を反映
  -> 再度プレビュー方法を案内

Phase E: 実装ハンドオフ
  -> プロトタイプ確定
  -> IMPLEMENTATION_PROMPT.md を生成
```

## Phase A: 要求の構造化

依頼者（PdM / エンジニア）から受け取る入力は以下のいずれか:

- ユーザーストーリー: 「〇〇として、△△したい。それによって□□できる。」
- 機能要求: 「商品詳細ページにレビュー投稿フォームを追加したい」
- UI イメージの説明: 「上部に通知バナーを表示。info/warning/error/success の 4 タイプ」
- Figma リンクやスクリーンショット

受け取った要求から以下のテンプレで構造化し、依頼者に確認する。

```text
■ コンポーネント名（仮）: Proto{PascalCase}   ※ Proto prefix 必須
■ 目的: [何を解決するか]
■ デバイス: [PC / SP / 両方]
■ 主要な機能:
  - [機能1]
  - [機能2]
■ 想定される状態:
  - [例: 初期表示 / 入力中 / 送信中 / 成功 / エラー / 空]
■ 使いそうな既存コンポーネント:
  - [既存コンポーネントを参照]
```

⚠️ **Proto prefix 必須**: プロトタイプのコンポーネント名・ディレクトリ名には必ず `Proto` を付ける。
理由: 既存プロジェクトに同名コンポーネントが存在する場合に混同を防ぐため。
例: `QuickOrder` をベースにしたプロトタイプ → `ProtoQuickOrder`

⚠️ この段階で依頼者と認識を合わせる。曖昧な点は推測せず質問する。

## Phase B: preset の確定

`project-config.md` から最低限以下を確認する。

- パス設定
- プレビュー方法
- コーディング規約
- 既存コンポーネント参照方法
- デザイントークン

現在は React + TypeScript + Storybook 系を対象とし、`variants/react-storybook.md` を使う。
preset は project 固有情報を持つ。

## Phase C: プロトタイプ生成

生成時の共通ルール:

- UI の見た目と操作感の確認が目的（完璧な実装ではない）
- Container、データ取得、API 連携は含めない（純粋な Presentational のみ）
- 簡易インタラクションに必要なローカル状態は許容する（フォーム入力→プレビュー反映、タブ切替、開閉トグル等。PdM とのすり合わせに必要な UI の動きは積極的に実装する）
- デザイントークンを優先し、値の直書きを避ける
- 既存コンポーネントが使えるなら積極的に使う（正しい import パス、正しい Props で）
- ファイル・ディレクトリ名に **Proto prefix** を必ず付ける

## Phase D: フィードバック反映

- 意図が曖昧な修正は質問してから直す
- 勝手にデザインを改善しない
- プレビュー確認に必要な状態は追加する

## Phase E: 実装ハンドオフ

プロトタイプ確定後に `IMPLEMENTATION_PROMPT.md` を出力する。

- 実際に生成したファイルを読み取って埋める
- 対話履歴の要求と制約を反映する
- stack に応じた template を使い分ける
- `IMPLEMENTATION_PROMPT.md` は実装計画（`implementation-plan`）の入力として渡す

## Phase F: 合意の記録（開発プロセス P2.5 として使う場合）

- プロトタイプ一式はマージしない proto branch に置き、Jira チケットからリンクする（短期コンテキスト。合意した操作感・状態は実装時に Storybook・テストとして永続化し、プロトタイプ自体は使い捨てる）
- 合意した UI の振る舞い・状態（初期・入力中・エラー・空など）を、受入れ条件の Delta として Jira チケットへコメントする。これが P2.5 の exit 条件

## 承認ポイント

- 要求構造化の確認時
- 初回プロトタイプ提示時
- 実装ハンドオフ出力前（PdM のプロトタイプ承認 = P2.5 のゲート）
