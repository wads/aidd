# 受入れ検証と昇格判定（P7）

ブランチレビューに加え、マージ前の P7 では以下を実施する。

## 1. 受入れ検証

- P2 の確定版 AC を 1 件ずつ検証し、AC-ID つきで手順と結果を PR コメントに記録する
- 検証手段タグが CI・テストの項目はゲート 1（テスト green）の結果を確認する
- 手動タグの項目（見た目・操作感・実機確認）はここで検証する
- 未検証項目が残る場合は「未検証項目」として理由つきで明示する

## 2. 整合確認

- diff が ADR・設計書と整合しているか確認する（設計と違う実装になっていないか）
- records_root 配下の `adr/`・`design/` に追加されたファイルの命名（`{連番}-short-title.md`、連番 4 桁 0 埋め）と連番重複を確認する

## 3. 短期コンテキストの昇格判定

短期層（PR 説明・コメント、Issue コメント）を走査し、以下に該当する内容を長期層へ昇格提案する。判定結果は「あり / なし」を必ず申告する。

- 将来の変更時に壊しやすい「守るべき振る舞い」が短期層にしかない → 設計書へ（`design-docs`）
- 選択肢を比較して捨てた判断が短期層にしかない → ADR へ（`adr`）

昇格は人間の承認を得てからマージ前に行う。

## 4. PR 説明の最終化

PR 説明に以下の必須セクションをそろえる。該当がない場合も「なし」と明記する。

- 受入れ条件⇄テスト対応表（AC-ID で参照）
- 技術判断: あり（ADR へのリンク + 設計意図・トレードオフ・影響範囲の要約）/ なし
- 短期コンテキストの昇格判定: あり（内容）/ なし
- 未検証項目: あり（内容と理由）/ なし

## 5. コンテキストスナップショット

`context-snapshot` skill でスナップショットを生成して Issue へ添付し、受入れ判断で人間が見るべき点（マーカー残存・未検証項目）を要約して提示する。

## 差し戻し

受入れ条件・品質条件を満たせないことが判明した場合は、その場で作り込まず、Intent Delta を Issue へ追記してから原因に応じたフェーズ（P2〜P5）へ戻る。
