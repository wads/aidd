# Multi-Agent Workflow

## 1. 前提の確定

- 何を判断したいか（topic）、答えを出したい問い（goal）を 1〜2 行で確定する
- 調査の前提事実（制約、既存環境、外せない要件）を明示する。これは全エージェントに同じく渡す共通の土台

## 2. 分割軸の決定（アンカリング回避の要）

各エージェントに別々の視点を割り当てる。同じ視点を複数に与えない。目的は「最初の案に全員が引きずられる」状態を避けること。

分割軸の例:

- 技術選定: 技術 A 担当 / 技術 B 担当 / 技術 C 担当（各担当は自分の技術を最善として深掘りする）
- システム設計: フロントエンド / バックエンド / インフラ
- 観点: 性能 / 保守性 / セキュリティ / 運用コスト
- 立場: 推進派 / 慎重派 / 既存維持派（同じ案への賛否を割る）

分割軸ごとに 1〜2 名、全体で 2〜6 名を目安にする。

## 3. 独立調査

- 各エージェントは互いの結論・調査中の情報を共有しない
- 各エージェントは目的・調査結果・推奨を構造化して報告する
- 「最初に出た案」「既存実装」「依頼者がこう言った」に合わせにいかない。割り当てられた視点から最善を探す

## 4. 統合（反証ベース）

- 結論が一致した点と、割れた点を分けて並べる
- 割れた点は、各案を擁護するのではなく互いに反証させる（「A の弱点は B 視点から見るとどう見えるか」）
- トレードオフ（何を取れば何を失うか）を曖昧にしない

## 5. 提示

- 推奨案・採用しなかった案・それぞれの理由を人間へ提示する
- 明確な優位案が無い場合は、無理に 1 案へ寄せず「割れている」と正直に報告し、人間の判断を仰ぐ（escalation）

## 6. 記録

- P3 から呼ばれた場合は、出力を `adr` に渡す（ADR の検討選択肢・決定理由・トレードオフの材料）
- 単体利用の場合は、必要に応じて ADR や decisions log に記録する

## ガイドライン

- ペルソナ・分割軸・人数は topic に応じて動的に決める
- 調査中のエージェント間で情報共有しない（独立性が崩れるとアンカリング回避の効果が消える）
- 結論を急がない。割れていることそのものが有用な情報
