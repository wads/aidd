# Multi-Agent Discussion

## 価値（なぜ単なる「比較して」と違うのか）

並行エージェントを起動する機能自体はビルトインにある。この skill の価値はメカニズムではなく **アンカリング回避の方法の規律** にある。

- 各エージェントは互いの結論を見ずに独立調査する（途中で情報共有しない）
- 各エージェントは最初に出た案・既存実装・依頼者の期待に引きずられないよう、別々の視点（分割軸）を割り当てられる
- 統合役は各案を擁護するのではなく、互いに反証させて差分を浮かび上がらせる

単一エージェントに「3 案比較して」と頼むと、最初に思いついた案に合わせた比較になりやすい。この skill はそれを構造的に防ぐ。

## いつ使うか

- 技術選定・設計方針を、偏りなく比較したいとき
- 既存実装や最初の思いつきに引きずられている疑いがあるとき
- P3 技術判断（`adr`）で、選択肢が割れる・影響が大きい判断の材料を集めるとき

## 入力

- `topic`: 何を判断したいか
- `perspectives`: 分割軸（省略時は topic から決める）
- `goal`: この調査で答えを出したい問い

## 出力

- 各視点の独立した調査結果
- 共通点と差分（特に結論が割れた点）
- 推奨案と、採用しなかった案・その理由

P3 から呼ばれた場合、この出力は `adr` が ADR の「検討した選択肢」「決定理由」「トレードオフ」を書く材料になる。単体利用の場合は、結論を必要に応じて ADR や decisions log に記録する。

## 扱う Intent

- Decision Intent の材料（判断そのものは人間が行い、`adr` が記録する）
