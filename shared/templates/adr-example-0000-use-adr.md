# [ADR-0000] Architecture Decision Records を採用する

## ステータス

承認済み

## 背景

AI駆動開発において、技術的な意思決定の経緯と理由を記録・追跡する必要がある。
コンテキストウィンドウを超えて開発を継続するため、過去の判断を後から参照できる仕組みが重要。

## 検討した選択肢

### 案1: ADR (Architecture Decision Records)

- 概要: MADR 形式で技術決定を個別ファイルに記録
- メリット: 標準的、検索しやすい、差分管理可能
- デメリット: ファイル数が増える

### 案2: 単一ファイルの決定ログ

- 概要: 1つのファイルに全決定を記録
- メリット: シンプル
- デメリット: 大きくなると管理困難、並行編集しにくい

## 決定内容

ADR (Architecture Decision Records) を MADR 形式で採用する。

## 設計意図

各決定が独立したファイルとして管理され、git diff で変更追跡が容易。
マルチAgent調査の結果や人間の判断を議論ログセクションに記録でき、
AI駆動開発における意思決定の透明性を確保する。

## トレードオフ

ADR 作成のオーバーヘッドが発生する。ただしテンプレート利用で軽減される。
ファイル数が増えるが、1決定1ファイルの粒度は検索性と差分管理の利点が上回る。

## 影響範囲

- 全プロジェクトの `docs/adr/` ディレクトリ
- `/dev` skill の Standard/Full モード（ADR 作成ステップ）
- PR レビュー時の設計判断確認フロー
