# Tutorial: AIDD で開発を始める

Claude Code を起動した後の具体的な進め方を、ステップバイステップで説明します。

## 前提

- Quick Start の手順でプロジェクトのセットアップが完了していること
- CLAUDE.md の Project Configuration（Test command 等）が設定済みであること

## Step 1: 要求を伝える

Claude Code を起動したら、作りたいものを自然言語で伝えます。

```
> TODOアプリのREST APIを作りたい。タスクのCRUD操作ができるようにしたい。
```

Claude が要求を整理し、確認を求めてきます。内容に問題がなければ承認してください。
要求が整理されると `docs/requirements/` に要求定義が作成されます。

## Step 2: 技術的な議論が必要な場合

技術選定やアーキテクチャの判断が必要な場合、複数Agentによる議論を実施できます。

```
> /multi-agent-discussion
```

**何が起きるか:**
1. researcher Agent が選択肢を調査
2. 各Agentが意見を出し合い、Pros/Consを整理
3. 合意された結論が `tasks/decisions-log.md` に記録される
4. 必要に応じて ADR（`docs/adr/`）が作成される

**あなたがやること:**
- Agentの結論を確認し、承認または修正を指示する
- 判断に迷う場合は追加の調査を依頼する

## Step 3: 要件定義と設計

要求が承認されると、Claude が要件定義 -> ADR -> 詳細設計と順に進めます。
各フェーズで承認を求められるので、内容を確認してください。

**あなたがやること:**
- 各フェーズの成果物を確認して承認する
- 不足や修正があればフィードバックする
- 「この部分をもっと詳しく」「ここは不要」等の指示を出す

## Step 4: 実装用の作業環境を作る

設計が固まったら、git worktree で実装用の隔離環境を作ります。

```
> /worktree-setup
```

**何が起きるか:**
1. feature名の確認（例: `feature/todo-api`）
2. git worktree + topic branch が作成される
3. 隔離された作業ディレクトリが準備される

**あなたがやること:**
- feature名を確認して承認する

## Step 5: TDD で実装する

実装は TDD サイクルで進めます。

```
> /tdd-cycle
```

**何が起きるか:**

サイクルが3つのフェーズで繰り返されます:

### Red（失敗するテストを書く）
- developer Agent がテストを作成
- テスト実行 -> 失敗を確認
- 「Red phase complete. Proceed to Green?」と確認される

### Green（テストを通す最小実装）
- テストを通す最小限のコードを実装
- テスト実行 -> 成功を確認
- 「Green phase complete. Proceed to Refactor?」と確認される

### Refactor（コード改善）
- 重複除去、命名改善等のリファクタリング
- テストが引き続き通ることを確認
- 「Refactor complete. Commit?」と確認される

**あなたがやること:**
- 各フェーズの結果を確認して次に進む
- テスト内容や実装方針に違和感があればフィードバックする
- commit タイミングを判断する

## Step 6: 振り返り

実装が完了したら、振り返りを実施します。

```
> /retrospective
```

**何が起きるか:**
1. developer と reviewer が KPT（Keep/Problem/Try）で議論
2. `tasks/retrospectives/YYYY-MM-DD-xxx.md` に記録
3. 汎用的な教訓が `tasks/lessons.md` に追記
4. プロセス改善提案がある場合は提示される

**あなたがやること:**
- KPT の内容を確認する
- プロセス改善提案があれば承認/却下を判断する

## Tips

### Skill を使わない自由な開発もOK

Skill はあくまでガイドです。慣れてきたら Skill を使わずに直接指示してもかまいません。

```
> この関数にバリデーションを追加して。テストも書いて。
```

### いつでも議論を挟める

開発中に判断に迷ったら、いつでも `/multi-agent-discussion` で議論できます。

```
> /multi-agent-discussion
> テーマ: エラーハンドリングの方針。独自エラー型を定義すべきか、標準のエラーで十分か。
```

### 教訓を活用する

`tasks/lessons.md` に蓄積された教訓は、Claude が自動的に参照します。
プロジェクトを重ねるほど、開発プロセスが改善されていきます。

### プロセスの全体像

```
[1.要求確認] --> 人間承認 --> [2.要件定義] --> 人間承認 --> [3.ADR作成]
    --> 人間承認 --> [4.詳細設計] --> 人間承認 --> [5.worktree作成]
    --> [6.TDD実装] --> 人間確認 --> [7.振り返り]
```

全フェーズを必ず踏む必要はありません。小さな修正なら Step 5 の `/tdd-cycle` だけでも十分です。
プロジェクトの規模や状況に合わせて、必要なフェーズだけを使ってください。
