# 設計叩き台: dev skill のフェーズ分解とコンテキスト契約

- Status: Draft（叩き台・議論用）
- Date: 2026-06-11
- 関連: [intent-driven-development.md](intent-driven-development.md), [ADR-0001](../adr/0001-restructure-skills-for-codex-and-claude.md)

## 目的

Issue 作成（`product-intent`）後の開発プロセスを工程（フェーズ）ごとに分解し、各フェーズについて以下を契約として定義する。

- AI エージェントと人間がそれぞれ何を行い、何を議論するか
- どの成果物・コードを作成するか
- どの品質・受入れ基準（exit 条件）で次フェーズへ進むか
- どのコンテキストを、どこに、どの寿命で残すか

この文書の合意後、各フェーズを skill として実装する。

## 非目的

- 全体を統括する orchestrator の設計（別途作成予定）
- 各 skill 本文の実装（この文書はその前段の合意形成。`product-intent` のタイプ対応改修を含む）
- worktree・branch の物理的な管理方法（`workspace-hygiene` と実行環境の責務。本プロセスは P1 の entry 条件として「クリーンな作業ツリー」を要求するだけで、その実現手段には関与しない）

## 前提（確認済み）

- 成果物の実体はプロダクト repo の `docs/` 配下に置く。playbook 側はテンプレートとルールのみを持つ
- スコープはIssue 作成後から retrospective まで
- 短期コンテキスト（実装方針・検証メモ等）は PR/Issue に残すだけでよい。repo に永続化するのは ADR・設計書
- 実行主体の AI エージェントは当面 Claude を想定するが、本文書の表記は特定エージェントに依存しない（以下「AI」と表記。Codex 等の他エージェントにも適用できる）

## 議論と決定の原則

本プロセスおよび派生するすべての skill の議論パートに適用する。

- AI は各フェーズの議論で、率直な意見・懸念・反論を提示する。人間の意向への同調や忖度を品質より優先しない
- 最終決定は常に人間が行う。ゲートの承認を AI が代行しない
- 意見が対立したまま決定された場合も決定に従う。ただし採用されなかった懸念は、ADR の「採用しなかった選択肢」や Intent Delta として記録してよい

## コンテキストの 2 層構造

intent-driven-development.md の Intent 分類を、寿命と置き場所で 2 層に整理する。

### 長期層（プロダクト repo、イミュータブル）

「なぜこの形なのか」を将来の AI と人間が復元するための記録。上書きせず、変更は新しい文書または Intent Delta で表す。

| 成果物 | Intent | 置き場所 |
|---|---|---|
| ADR | Decision Intent | `docs/adr/{連番}-{Issue 番号}-short-title.md`（例: `00042-#123-cache-strategy.md`） |
| 設計書 | Design Intent | `docs/design/{連番}-{Issue 番号}-short-title.md` |
| テストコード | Verification Intent | テストディレクトリ（コードとして永続） |

### 短期層（PR / GitHub Issue、使い捨て）

該当タスクを正しく完了させるための作業コンテキスト。マージ・クローズ後はメンテナンスしない（検索可能なら十分）。

| 成果物 | Intent | 置き場所 |
|---|---|---|
| 確定版受入れ条件・品質条件 | Product Intent への Intent Delta | Issue コメント |
| プロトタイプ一式・`IMPLEMENTATION_PROMPT.md` | Design Intent（UI） | マージしない proto branch（GitHub Issue からリンク） |
| 実装方針 | Execution Intent | draft PR の説明文 |
| 受入れ条件⇄テスト対応表 | Verification Intent | PR 説明文 |
| 検証メモ（手順と結果） | Verification Intent | PR コメント |
| コンテキストスナップショット（HTML） | 全 Intent の生成ビュー | GitHub Issue添付（再生成可能・保守しない） |
| 振り返り | Learning Intent | Issue コメント（採用された改善は playbook への PR） |

### 短期コンテキストの昇格ルール

短期層の内容は原則使い捨てだが、P7（検証・レビュー）と P8（振り返り）で次を判定し、該当すればマージ/クローズ前に長期層へ昇格（短期コンテキストの昇格）させる。

- 将来の変更時に壊しやすい「守るべき振る舞い」が短期層にしか書かれていない → 設計書へ
- 選択肢を比較して捨てた判断が短期層にしか書かれていない → ADR へ

判定は AI が提案し、人間が承認する。

### 相互リンク規約

索引は GitHub Issue が担うため、全成果物に Issue 番号（`#{番号}` 形式。`#123`、`#11`、`#4356` など）を機械的に埋め込む。本規約の例では `#123` を使うが、特定のリポジトリ・プロジェクトには依存しない。Issue を別リポジトリ（コンテキストハブ）へ集約する場合は Binding の `issue_repo` を起点に参照する。

- ADR・設計書のファイル名: `{連番}-{Issue 番号}-short-title.md`（例: `00042-#123-cache-strategy.md`）。連番は 5 桁 0 埋めで、`docs/adr/` と `docs/design/` それぞれ独立に増やす
- 同一Issue から複数の ADR が出る場合は連番と short-title で区別する
- short-title は小文字ケバブケースとする（大文字のIssue 番号部分と機械的に区別できるようにするため）
- Issue に紐づかない文書（横断的な技術判断など）はIssue 番号部分を省略する（例: `00043-introduce-feature-flags.md`）
- 並行ブランチで連番が衝突した場合は、後にマージする側がリネームする（P7 のチェック項目に連番重複の検出を含める）
- branch 名: `feat/#123-short-desc`
- PR タイトル: `#123: 変更の要約`
- Issue には ADR・設計書へのリンクをコメントで貼る

連番はディレクトリ内の並び順（読む順序のヒント）のためのもの。文書の新旧・置き換え関係の正は、置き換える側の文書に明記する「Replaces: {旧文書}」とする。

### PR 説明の必須セクション

PR 説明には以下のセクションを必須とする。該当がない場合も省略せず「なし」と明記する（暗黙の省略を構造的に不可能にするため）。

- 受入れ条件⇄テスト対応表（AC-ID で参照）
- 技術判断: あり（ADR へのリンク）/ なし
- 短期コンテキストの昇格判定: あり（内容）/ なし
- 未検証項目: あり（内容と理由）/ なし

既存の PR テンプレートを持つプロダクトでは、テンプレートを置き換えず必須セクションを既存テンプレートへ組み込む。組み込み方はプロダクトごとの preset で吸収する。

## フェーズ全体像

```text
product-intent（済）
  → P1 着手 → P2 要求整理 ⇄ P2.5 プロトタイピング（UI 変更時のみ）
  → P3 技術判断 ⇄ P4 設計 → P5 実装計画
  → P6 実装 → P7 検証・レビュー → P8 振り返り
```

P3 と P4 は反復可能とする（設計の途中で新しい技術判断が生まれることは普通にある）。P2 と P2.5 も同様に反復する（プロトタイプの合意が受入れ条件を変える）。

### 受入れ条件・品質条件のトレース

受入れ条件・品質条件は次の経路でコードに到達し、2 段のゲートで検証される。

1. P2 で確定し、ID（`AC-n` / `QC-n`）を振る（Issue コメント）。P2.5 を実施した場合は UI の合意を Delta として反映する
2. P5 で各実装ステップに対応付ける（PR 説明）
3. P6 で `tdd-cycle` によりテストへ変換する（受入れ条件⇄テスト対応表）。設計書の「守るべき振る舞い」も同様にテスト化する
4. **ゲート 1（P6 exit、主に機械）**: 全 AC に対応するテストが green（対応の完全性は AC-ID で機械チェック可能）、lint・型・カバレッジ等の機械チェック可能な品質条件を満たす（検証手段タグが CI・テストの項目）
5. **ゲート 2（P7、人間 + AI）**: テストで表現できない条件（見た目・操作感・実機確認）の検証（検証手段タグが手動の項目）、diff と ADR・設計書の整合確認、人間レビューと受入れ判断

### 差し戻し（ゲートで条件を満たせない場合）

後段のゲート（特に P6・P7）で受入れ条件・品質条件を満たせないことが判明した場合は、その場で作り込まず、原因に応じたフェーズへ戻る。

| 満たせない原因 | 戻り先 |
|---|---|
| 受入れ条件が曖昧・矛盾、テストとして表現できない | P2（プロダクト判断が必要なら `product-intent` の確認ループへ） |
| 合意した UI・操作感が実現できない、または合意が崩れた | P2.5 |
| 技術選定の前提が崩れた | P3（既存 ADR は書き換えず、置き換える ADR を追加） |
| 設計の責務分割・フローが成立しない | P4 |
| 実装順序・粒度だけの問題 | P5 |

戻る際のルールは 1 つだけ: **戻る理由を Intent Delta として Issue へ追記してから戻る**。サイレントに後段で辻褄を合わせない。

| フェーズ | AI の作業 | 人間の作業（承認 = ゲート） | 成果物 → 置き場所 |
|---|---|---|---|
| P1 着手 | 作業ツリー整理、Issue読解、ルート特定 | ルート承認 | branch、ルート判断 → Issue コメント |
| P2 要求整理 | 曖昧さ・エッジケース指摘、品質条件提案 | 不明点回答、品質条件承認 | 確定版受入れ条件・品質条件 → Issue コメント |
| P2.5 プロトタイピング | プロトタイプ生成、PdM とすり合わせ | PdM がプロトタイプ承認 | プロトタイプ → proto branch、UI 条件 Delta → GitHub Issue |
| P3 技術判断 | 選択肢整理、推奨と理由 | 採用判断 | ADR → `docs/adr/` |
| P4 設計 | 責務分割、フロー、例外設計 | 設計承認 | 設計書 → `docs/design/` |
| P5 実装計画 | 実装順序、変更/不変更範囲、検証方針 | 計画承認 | 実装方針 → draft PR 説明 |
| P6 実装 | tdd-cycle、テスト⇄受入れ条件対応付け | 確認ポイント承認、逸脱判断 | コード + テスト、対応表 → PR 説明 |
| P7 検証・レビュー | self-review、受入れ検証、昇格提案、スナップショット生成 | PR レビュー、受入れ判断、マージ | 検証メモ → PR コメント、スナップショット → GitHub Issue 添付 |
| P8 振り返り | KPT、意図伝達の失敗分析、改善提案 | 改善の採否 | 振り返り → GitHub Issue、改善 → playbook PR |

## タイプ別ルート

どのフェーズを実施するかは、変更規模（旧 Lite / Standard / Full）ではなくIssue のタイプで決める。タイプはIssue 作成時（`product-intent`）の必須入力とし、GitHub ラベル（`type-*`）として付与する。P1 はラベルを読むだけでルートが決まる。

規模・リスクによる深さの調整は、ルールの軸を増やすのではなく、判断駆動の成果物作成（ADR・設計書は判断が生じたときだけ）と escalation（想定外が出たら停止して人間判断）が担う。**タイプ × 規模のマトリクスは作らない**。

| タイプ | 内容 | 受入れ条件の核 | ラベル / 課題タイプ |
|---|---|---|---|
| feature | 機能追加・仕様変更 | ユーザーに見える振る舞い | `type-feature` / Story |
| bugfix | 欠陥修正 | 再現手順で直っている + 回帰なし | `type-bugfix` / Bug |
| hotfix | 緊急の欠陥修正 | bugfix と同じ + 事後記録 | `type-hotfix` / Bug |
| refactoring | 外部振る舞い不変の内部改善 | 既存の振る舞いが変わらないこと | `type-refactoring` / Task |
| chore | 定型作業（依存更新、設定変更、軽微な文言） | 作業完了の機械的確認（CI green 等） | `type-chore` / Task |
| spike | 調査・検証（コードは成果物にしない） | 問いに答えが出ていること | `type-spike` / Task |

ラベルに `type-` prefix を付けるのは、既存の運用ラベル（`商品ラインナップ`、`OSL-UI` 等）と機械的に区別するためである。

| フェーズ | feature | bugfix | refactoring | chore | spike |
|---|---|---|---|---|---|
| P1 着手 | ✓ | ✓ | ✓ | ✓ | ✓ |
| P2 要求整理 | ✓ | ✓（再現条件・回帰範囲） | ✓（不変条件・対象範囲） | 簡易 | ✓（問いの定義） |
| P2.5 プロトタイピング | UI 変更時 | − | − | − | − |
| P3 技術判断 | 判断駆動 | 判断駆動（稀） | 判断駆動 | − | 結論を ADR 化しうる |
| P4 設計 | 判断駆動 | − | 判断駆動 | − | − |
| P5 実装計画 | ✓ | 簡易（PR 説明 1 段落） | ✓ | − | − |
| P6 実装 | ✓ | ✓（再現テスト先行） | ✓（既存テスト green 維持） | ✓ | 使い捨てコード可（マージしない） |
| P7 検証・レビュー | ✓ | ✓ | ✓ | 簡易 | 報告のレビュー |
| P8 振り返り | 推奨 | 任意 | 任意 | − | 必須（学びが主成果物） |

hotfix は bugfix の緊急変形とする。ゲートを P6・P7 の最小構成に圧縮して先に直し、事後に P8 と記録（なぜ緊急対応したかの Intent Delta、必要なら ADR）を必須にする。事後記録漏れの専用監査は設けない（「Issue がなければリリースできない」社内ルールが入口で防ぐため）。

運用ルール:

- タイプが混在するIssue（機能追加 + ついでのリファクタリング等）は分割する（「変更を小さく刻む」の適用）。分割できない場合は重い方（feature）のルートに乗せる
- タイプはIssue 作成時に決める（`product-intent` が必須入力として確認する。迷う場合は AI が推測せず人間に確認する）
- タイプラベルのないIssue には着手しない。P1 で検出したらIssue入口へ差し戻す
- 着手後にタイプが誤りと判明した場合はラベルを修正し、P1 からルートを引き直す
- ルートはフェーズの実施有無を決めるだけで、成果物の深さは決めない。実施するフェーズの中で何をどこまで書くかは判断駆動の原則に従う
- 実施しないフェーズの責務が消えるわけではない。例えば chore で「守るべき振る舞い」に触れたなら、P7 の短期コンテキストの昇格ルールで設計書化する

### タイプ別Issueテンプレート（骨子）

feature は現行の Product Intent テンプレートをそのまま使う。他タイプは必須項目を絞った軽量テンプレートとし、詳細は `product-intent` 改修時に確定する。エンジニア起点のIssue（bugfix / chore / spike）は確認ループを最小にし、記入が数分で終わることを設計条件とする。

- **bugfix**: 再現手順、期待する挙動、実際の挙動、影響範囲（誰が・どこで）、回帰の懸念箇所
- **hotfix**: bugfix と同じ + 緊急性の根拠。Issue 作成が事後になることを許容する（先に直し、Issue と記録を事後に整える）
- **refactoring**: 動機（なぜ今やるか）、対象範囲、不変条件（守るべき振る舞い）、対象外
- **chore**: 作業内容、完了の確認方法
- **spike**: 答えるべき問い、背景、time-box、成果物の形（ADR / 報告メモ）

## フェーズ詳細

### P1 着手

- **entry**: 品質チェック済み GitHub Issue（Product Intent）
- **AI**: `workspace-hygiene` で作業ツリーを整理し topic branch を作成。Issue を読解し、自分の言葉で要求を言い直す。タイプラベル（`type-*`）からルートを特定し、適用するフェーズ構成を提示する（ラベルのないIssue は進めず、Issue入口へ差し戻す）。feature の場合は UI に見える変更を含むかを判断し、P2.5（プロトタイピング）の実施要否を提案する
- **人間**: 言い直しのずれを修正し、ルートと P2.5 実施要否を承認する
- **議論の中心**: スコープの境界とリスク認識のすり合わせ
- **exit**: 人間が「AI の理解はずれていない」と確認し、ルートが合意されている
- **残すコンテキスト**: ルートと判断理由を 1〜2 行で Issue コメントへ

### P2 要求整理

- **entry**: ルート合意済み
- **AI**: Product Intent を開発で使える粒度へ具体化する。(1) 受入れ条件の曖昧・矛盾・漏れ（エッジケース）を指摘する。(2) 受入れ条件から導かれる技術品質条件（テスト範囲、性能、互換性、セキュリティ、観測性）を提案する。各受入れ条件・守るべき振る舞いをどのテストレベル（unit / integration / E2E）で検証するかを割り当て、各品質条件には検証手段タグ（CI / テスト / 手動）を付ける（P6・P7 のどちらが何を検証するかを機械的に決めるため）。観点は `requirements` skill の標準質問リストを走査し、該当する観点だけを測定可能な基準として書き、該当しない観点は「他は対象外」の 1 行に畳む。確定した受入れ条件・品質条件には ID（`AC-1`…、`QC-1`…）を振り、以後の実装計画・テスト・対応表・検証記録はこの ID で参照する（対応の完全性を機械チェック可能にするため）。各項目には信頼性マーカーを付ける — 🔵 根拠あり（出典明記）/ 🟡 妥当な推定 / 🔴 根拠なし・要確認。推測をマーカーなしで埋め込まない
- **人間**: 曖昧点に回答し、品質条件を承認する。プロダクト判断が必要な場合は PdM へ差し戻す（`product-intent` の確認ループへ戻る）
- **議論の中心**: 「完了とは何か」の確定。受入れ条件（ユーザー視点）と品質条件（技術視点）を分けて扱う
- **exit**: 全受入れ条件が「検証手順を書ける」粒度になっている。🟡・🔴 マーカーが残っていない（人間が確認して 🔵 にするか、対象外として畳む）
- **残すコンテキスト**: 確定版の受入れ条件・品質条件を Issue コメントへ（Issue本文は書き換えず Intent Delta として追記）
- **Intent**: Product Intent（入力）/ Intent Delta（追記）

### P2.5 プロトタイピング（UI に見える変更があるときのみ）

- **entry**: P2 の受入れ条件（暫定版でよい）。UI に見える変更を含まない場合はスキップ
- **AI**: `ui-prototyping` でプロトタイプを生成し、PdM と見た目・操作感をすり合わせる。確定後 `IMPLEMENTATION_PROMPT.md` を生成する
- **人間**: PdM がプロトタイプを承認する（「コレジャナイ」をここで潰す）
- **議論の中心**: 見た目・操作感・状態（初期・入力中・エラー・空など）の合意
- **exit**: 合意した UI の振る舞い・状態が受入れ条件へ反映されている（Issue へ Delta 追記）
- **残すコンテキスト**: プロトタイプ一式と `IMPLEMENTATION_PROMPT.md` → マージしない proto branch（GitHub Issue からリンク、短期層）。UI 受入れ条件の Delta → Issue コメント。合意した操作感・状態は P6 で Storybook・テストとして永続化し、プロトタイプ自体は使い捨てる
- **Intent**: Product Intent（Delta 追記）/ Design Intent（UI）

### P3 技術判断

- **entry**: 確定済み要求。選択肢のある技術判断が存在する場合のみ実施
- **AI**: 選択肢を整理し（選択肢が割れる、または影響が大きい場合は `multi-agent-discussion` を使用）、推奨案・理由・トレードオフ・採用しなかった選択肢を提示する。明確な優位案がなければ escalation する
- **人間**: 採用を判断する
- **議論の中心**: トレードオフの重み付け（何を優先し何を捨てるか）
- **exit**: 採用しなかった選択肢とその理由が ADR に書かれている
- **残すコンテキスト**: ADR → `docs/adr/{連番}-{Issue 番号}-short-title.md`（イミュータブル。判断が変わったら新しい ADR で置き換える）
- **Intent**: Decision Intent

### P4 設計

- **entry**: 技術判断が確定（または判断不要と合意）
- **AI**: 目的・非目的、責務分割、データと状態の扱い、主要フロー、例外ケース、守るべき振る舞い、未確定事項を設計書にまとめる。各記述に信頼性マーカー（🔵/🟡/🔴）を付け、推定で書いた箇所を明示する。技術選定の理由は ADR へ寄せ、重複させない
- **人間**: 設計を承認する
- **議論の中心**: 責務の置き場所、既存設計との整合、例外ケースの扱い
- **exit**: 受入れ条件の各項目について「設計のどこで満たされるか」を対応付けられる
- **残すコンテキスト**: 設計書 → `docs/design/{連番}-{Issue 番号}-short-title.md`
- **Intent**: Design Intent

### P5 実装計画

- **entry**: 設計承認済み（P3・P4 を実施しないルートでは P2 から直行）
- **AI**: 実装順序とその理由、変更対象と触らない範囲（それぞれ理由つき）、リスク、検証方針、人間確認ポイントを計画にまとめる。各ステップに TDD / DIRECT の区分を付け、DIRECT（テストで駆動できない作業: 環境構築・設定変更など）には実施手順と検証手順を必須にする
- **人間**: 計画を承認する
- **議論の中心**: 刻み方（1 ステップが PR レビュー可能な粒度か）と確認ポイントの配置
- **exit**: 各ステップが受入れ条件・設計のどの項目に対応するか（AC-ID 参照）と、TDD / DIRECT の区分が明示されている
- **残すコンテキスト**: 実装方針 → draft PR の説明文（使い捨て。判断理由のうち長期価値のあるものは P7 の短期コンテキストの昇格で記録化）
- **Intent**: Execution Intent

### P6 実装

- **entry**: 計画承認済み
- **AI**: `tdd-cycle` で実装する。テスト名・ケースを受入れ条件と対応付け、テストレベルは P2 で割り当てたものに従う。計画からの逸脱（スコープ変更、想定外の問題、新しい技術判断）が生じたら停止して人間へ確認する（現行 `escalation.md` を踏襲）
- **人間**: 確認ポイントで承認し、逸脱時の判断を行う
- **exit**: 全受入れ条件に対応するテストが green。品質条件（P2 で確定したもの）を満たしている
- **残すコンテキスト**: コードとテスト（テスト自体が Verification Intent の永続記録）。受入れ条件⇄テストの対応表を PR 説明へ
- **Intent**: Verification Intent

### P7 検証・レビュー

- **entry**: 実装完了
- **AI**: (1) `review` skill で self-review。(2) 受入れ条件を 1 件ずつ検証し、手順と結果を記録する。(3) PR 説明を最終化する（ADR の設計意図・トレードオフ・影響範囲を含める — common.md の既存ルール）。(4) 短期コンテキストの昇格ルールに従い、長期層へ移すべき内容を提案する。(5) `context-snapshot` で、この開発単位の最新の意図と検証状態を人間向け HTML スナップショットとして生成し、受入れ判断の材料として提示する。スナップショットには信頼性マーカーの残存サマリー（未解決の 🟡・🔴 項目）を含め、受入れ判断で人間が見るべき注意点として明示する
- **人間**: PR をレビューし、受入れを判断し、短期コンテキストの昇格提案を承認し、マージする。マージ後のリリース検証もここに含める
- **exit**: 全 AC の検証結果が AC-ID 付きで記録されている。未検証項目が残る場合は明示されている
- **残すコンテキスト**: 検証メモ → PR コメント。コンテキストスナップショット（HTML、生成日付き） → GitHub Issue 添付。短期コンテキストから昇格した ADR・設計書（あれば）
- **Intent**: Verification Intent（記録）/ 全 Intent との整合確認

### P8 振り返り

- **entry**: マージ・リリース検証後
- **AI**: `retrospective`（KPT）を実施する。特に「意図が後工程へ伝わらなかった点」「認知負債になりかけた点」を抽出し、playbook の skill・テンプレートへの具体的な改善を提案する
- **人間**: 改善の採否を判断する
- **exit**: 採用された改善が playbook への PR またはIssue になっている
- **残すコンテキスト**: 振り返り → Issue コメント。採用改善 → playbook への PR
- **Intent**: Learning Intent

## 論点（議論したい点）

叩き台作成者（AI）の意見を含む。賛否を議論したい。

1. **長期コンテキストに薄い可変レイヤが必要ではないか。** イミュータブルな ADR + Delta の積み上げは判断履歴の保全には優れるが、「いまの設計がどうなっているか」の把握には不向き（Delta が 20 個になった時点で順読みは破綻する）。意見: 判断記録（immutable）とは別に、living な現在仕様ドキュメント（例: `docs/spec/`、改訂時は旧版を archive）を持つべき。これは intent-driven-development.md の「1 つの文書へ集約しない」「上書きしない」方針と部分的に衝突するため、原則の例外として明文化が必要。**結論（2026-06-12）**: 人間が維持する living doc は持たない。現在状態はコード・テスト・不変記録からその場で生成する view として扱い、P7 の `context-snapshot`（HTML スナップショット）がその実体。スナップショットは再生成可能な使い捨てで、不変記録と食い違った場合は記録側を正とする。
2. **Intent Context ファイルは作らず、GitHub Issueを索引にすべきではないか。** intent-driven-development.md は repo 内に Intent Context ファイルを置く想定だが、「短期コンテキストは PR/Issue に残るだけでよい」という今回の方針と運用負荷の観点から、Issue に ADR・設計書・PR へのリンクを貼ることで索引の役割を果たす方が軽い。意見: repo 内 Intent Context ファイルは作らない。採用する場合 intent-driven-development.md の改訂が必要。**結論（2026-06-12）**: repo 内 Intent Context ファイルは作らず、GitHub Issueを索引にする（相互リンク規約を参照）。intent-driven-development.md の改訂は次のステップで行う。
3. **mode 判断の責務はどこに置くか。** 案 A: orchestrator（P1）が判断する。案 B: `requirements`（P2）で要求を具体化した後でないと規模は判断できないので P2 の出力とする。意見: 初期判断は P1、P2 完了時に再評価（mode の引き上げのみ許可、引き下げは人間判断）のハイブリッド。**結論（2026-06-12）**: mode（Lite / Standard / Full）という規模軸自体を廃止し、Issueタイプ別ルート方式へ置き換える（「タイプ別ルート」参照）。ルートは P1 で課題タイプから機械的に特定し、迷う場合は推測せず都度人間に確認する。深さの調整は判断駆動の成果物作成と escalation が担う。
4. **品質条件の標準セットを playbook 側に持つか。** P2 で毎回ゼロから品質条件を考えるのは不安定。意見: playbook に標準チェックリスト（テスト範囲、性能、互換性、セキュリティ、観測性）を置き、プロダクト repo 側で preset として上書き可能にする（stack 差分の preset/variant ルールと整合）。テストレベルの配分方針（ピラミッド型・トロフィー型など）も、比率の目標ではなく「どの種類の振る舞いをどのレベルで検証するか」の規則として preset に持たせる。**結論（2026-06-12）**: 採用。標準質問リスト（テスト範囲・レベル、性能、互換性、セキュリティ、観測性）を `requirements` skill 配下に置き、該当する観点だけを測定可能な基準として記載（該当しない観点は「他は対象外」の 1 行）、各品質条件に検証手段タグ（CI / テスト / 手動）を付ける。プロダクトごとの上書き preset は playbook 側の presets/ に置く（ui-prototyping と同じ機構）。

## 機械化候補（CI / hook）

以下は規律（AI が指示に従うこと）に頼らず機械的に強制できる項目の一覧。実装は各プロダクト repo の CI・hook で行うが、何を機械化すべきかの一覧は playbook を正とする。

- 着手時: タイプラベル（`type-*`）の存在、作業ツリーがクリーンであること
- 命名: branch 名・PR タイトルのIssue 番号形式、`docs/adr/`・`docs/design/` のファイル名形式と連番重複
- PR: 必須セクション（対応表・技術判断・昇格判定・未検証項目）の存在
- 対応の完全性: 全 AC-ID が対応表に出現し、対応するテストが存在すること
- マーカー残存: ゲート通過時、成果物に 🟡・🔴 の信頼性マーカーが残っていないこと（grep で検出可能）
- ゲート 1: テスト・lint・型・カバレッジ（既存 CI）
