# Development Workflow

AI駆動開発の全体フロー定義。タスクの規模・リスクに応じて3つのモードから選択する。

## Process Modes

### Lite（デフォルト）

日常的な機能追加・バグ修正に使用する最小プロセス。

```
[要求確認] --> 人間承認 --> [TDD実装] --> 人間確認
    --> (判断があればADR記録) --> 完了 --> 「振り返りしますか？」(任意)
```

- **要求確認**: 要求内容を整理し、人間に確認。ズレ防止が目的
- **TDD実装**: `/tdd-cycle` でRed-Green-Refactorサイクルを実行
- **ADR記録**: 技術的な判断があった場合、`docs/adr/` に軽量ADRを記録
- **振り返り**: タスク完了時に実施するか確認。希望すれば `/retrospective` を実行

### Standard

複数ファイルにまたがる中規模機能、技術選定を伴う開発に使用。

```
[要求確認] --> 人間承認 --> [要件定義] --> 人間承認
    --> [マルチAgent調査] --> [ADR作成] --> 人間承認
    --> [TDD実装] --> 人間確認 --> 完了 --> 「振り返りしますか？」(任意)
```

- Lite の全ステップに加え:
- **要件定義**: 要求を具体的な要件に分解。`docs/specifications/` に作成
- **マルチAgent調査**: `/multi-agent-discussion` で技術選択肢を並行調査
- **ADR作成**: 調査結果を踏まえた技術決定を記録

### Full

新規アーキテクチャ導入、大規模リファクタリング等に使用。

```
[要求確認] --> 人間承認 --> [要件定義] --> 人間承認
    --> [マルチAgent調査] --> [ADR作成] --> 人間承認
    --> [詳細設計] --> 人間承認 --> [TDD実装] --> 人間確認
    --> 完了 --> 「振り返りしますか？」(任意)
```

- Standard の全ステップに加え:
- **詳細設計**: 実装方針を設計し、`docs/designs/` に設計書を作成。レビューを実施
- **要求定義**: `docs/requirements/` にフォーマルな要求定義を作成

## Mode Selection Guide

| 判断基準 | Lite | Standard | Full |
|---------|------|----------|------|
| 変更規模 | 1-2ファイル | 複数ファイル | モジュール横断 |
| 技術的な判断 | 既知のパターン | 選択肢がある | 新しいアーキテクチャ |
| リスク | 低い | 中程度 | 高い |
| 例 | バグ修正、小機能追加 | 新機能、API追加 | DB変更、認証基盤刷新 |

## Principles

- モードに応じて適切なチェックポイントを設置する
- 必要に応じてマルチAgent調査を実施し、人間に選択肢を提示する
- 最終判断は常に人間が行う
- 小さく・段階的に進める（30行以下の変更を基本単位とする）
