# 人間向けマニュアル

AIDD playbook の開発プロセスを、**人間（開発者・PdM・レビュアー）の視点** からまとめたマニュアル。「各フェーズで自分は何を判断し、何を承認し、どこを見ればよいか」を解説する。

最短の操作手順は [チュートリアル](tutorial.md)、設計の詳細は [開発フェーズとコンテキスト契約](design/0001-dev-phase-decomposition.md) を参照。

## 大前提

- **最終決定は常に人間が行う。** AI はフェーズごとに作業し、節目で承認を求める。承認ゲートを AI が代行することはない
- **AI は率直に意見・懸念・反論を述べる。** 同調しない。納得できなければ議論してよい。意見が割れたまま人間が決めた場合、AI はその決定に従い、必要なら反対意見を記録に残す
- **ドキュメントは「書くべき判断が生じたとき」だけ作る。** ADR・設計書が無いことは手抜きではなく、判断が無かったということ
- **困ったら止まる。** 想定外・前提崩れが起きたら AI は作り込まず停止して確認を求める

## タイプ別ルート

Issue のタイプ（`product-intent` で確定し `type-*` ラベルで管理）によって、通るフェーズが決まる。

| タイプ | 主に通るフェーズ | 受入れ条件の核 |
|---|---|---|
| feature | P2 →（UI なら P2.5）→（判断あれば P3/P4）→ P5 → P6 → P7 → P8 | ユーザーに見える振る舞い |
| bugfix | P2（再現条件）→ P5 簡易 → P6（再現テスト先行）→ P7 | 再現手順で直っている + 回帰なし |
| hotfix | P6 → P7 を先に。P8 と記録は事後に必須 | bugfix と同じ + 事後記録 |
| refactoring | P2（不変条件）→ P5 → P6（既存テスト green 維持）→ P7 | 既存の振る舞いが変わらないこと |
| chore | P2 簡易 → P6 → P7 簡易 | CI green 等の機械的確認 |
| spike | P2（問いの定義）→ 調査 → 報告 → P8 必須 | 問いに答えが出ていること |

- タイプが混在する依頼は分割する。分割できなければ重い方（feature）のルートに乗る
- タイプの誤りに後から気づいたら、ラベルを直して P1 からルートを引き直す

## フェーズごとの「あなたがやること」

### 入口: Issue 作成（product-intent）

- **AI**: 依頼からタイプを提案し、タイプ別テンプレートで構造化する
- **あなた**: タイプの提案を確認する。feature は受入れ条件まで、それ以外は軽量テンプレート（数分）。内容を承認するとIssue が作られる
- **見るもの**: Issue本文（Product Intent）

### P1 着手（dev）

- **AI**: 作業ツリーを整理して branch を作り、要求を自分の言葉で言い直し、ルートを提示する
- **あなた**: 言い直しに認識のズレがないか確認する。ルートと、feature なら P2.5（プロトタイピング）の要否を承認する
- **判断のコツ**: ここでズレを潰すのが一番安い。違和感があれば遠慮なく指摘する

### P2 要求整理（requirements）

- **AI**: 受入れ条件（AC）の曖昧・矛盾・漏れを指摘し、品質条件（QC）を提案する。各項目に ID と信頼性マーカー（🔵/🟡/🔴）を付ける
- **あなた**: 🟡（推定）🔴（要確認）の項目に答えて確定させる。「完了とは何か」を承認する
- **見るもの**: Issue コメントの「確定版 受入れ条件・品質条件」
- **判断のコツ**: 🟡🔴 が残ったまま先に進めない。ここが曖昧だと後工程が AI の推測で埋まる。プロダクト判断が要るものは PdM に振る

### P2.5 プロトタイピング（ui-prototyping、UI 変更時のみ）

- **AI**: プロトタイプを生成し、状態（初期・入力中・エラー・空など）を見せる
- **あなた（PdM）**: 見た目・操作感を確認し、「コレジャナイ」をここで潰す。承認する
- **見るもの**: プロトタイプのプレビュー。合意内容は Issue に受入れ条件の追記として残る

### P3 技術判断（adr、判断があるときだけ）

- **AI**: 選択肢・推奨案・トレードオフ・捨てる案を提示する。迷う場合は止めて相談する
- **あなた**: 何を優先し何を捨てるかを判断する
- **見るもの**: `docs/adr/` に追加される ADR

### P4 設計（design-docs、判断があるときだけ）

- **AI**: 責務分割・主要フロー・例外・守るべき振る舞いをまとめる
- **あなた**: 責務の置き場所、既存設計との整合を確認して承認する
- **見るもの**: `docs/design/` に追加される設計書

### P5 実装計画（implementation-plan）

- **AI**: 実装順序、変更/不変更範囲、各ステップの TDD / DIRECT 区分、確認ポイントを計画する
- **あなた**: 刻み方（1 ステップが PR レビューできる粒度か）と確認ポイントの位置を承認する
- **見るもの**: draft PR の説明文

### P6 実装（tdd-cycle）

- **AI**: テストファーストで実装し、各テストを AC-ID に対応付ける。計画から外れたら止まる
- **あなた**: 確認ポイントで結果を見て進める。逸脱が起きたら判断する
- **ゲート**: 全 AC に対応するテストが green、lint・型などが通る（主に CI が機械チェック）

### P7 検証・レビュー（review + context-snapshot）

- **AI**: self-review、AC を 1 件ずつ検証、PR 説明の最終化、長期に残すべき判断の昇格提案、HTML スナップショット生成
- **あなた**: スナップショットと PR を見て受入れを判断し、昇格提案を承認し、マージする
- **見るもの**: Issue に添付される **コンテキストスナップショット（HTML）** がこのフェーズの主役。未検証項目・未解決マーカーが冒頭にまとまっている
- **判断のコツ**: スナップショットの「注意が必要な点」を最初に見る。ここが空なら安心して受け入れられる

### P8 振り返り（retrospective）

- **AI**: KPT で振り返り、特に「意図が後工程へ伝わらなかった点」を抽出し、playbook への改善を提案する
- **あなた**: 改善の採否を決める。採用したものは playbook への PR になる

## ゲートと差し戻しの考え方

- **ゲート** = 人間の承認（または機械チェック）。飛ばさない。軽量化しても P2・P6・P7 のゲートは省略しない
- **差し戻し**: 後工程で条件を満たせないと分かったら、その場で辻褄を合わせず原因のフェーズへ戻る。戻る理由を Issue に Intent Delta として残してから戻る
  - 受入れ条件が曖昧 → P2 / UI 合意の崩れ → P2.5 / 技術前提の崩れ → P3 / 設計不成立 → P4 / 順序・粒度だけ → P5

## コンテキストはどこにあるか

「あの判断どこで決めたっけ」を探すときの早見表。索引は GitHub Issue。

| 知りたいこと | 見る場所 |
|---|---|
| 何を作るか・なぜ | GitHub Issue本文（Product Intent） |
| 確定した受入れ条件・品質条件 | Issue コメント（P2 の確定版） |
| 途中での方針変更 | Issue コメントの Intent Delta |
| 技術判断の理由・捨てた案 | `docs/adr/{連番}-{Issue 番号}-*.md` |
| 設計・守るべき振る舞い | `docs/design/{連番}-{Issue 番号}-*.md` |
| 実装方針・テスト対応 | PR の説明文 |
| 検証結果 | PR コメント |
| 今の全体像（1 枚） | GitHub Issue 添付のコンテキストスナップショット（HTML） |

短期（Issue コメント・PR）はマージ後メンテしない。長期（ADR・設計書・テスト）が「なぜこの形か」の正。両者が食い違ったら長期側を正とする。

## 信頼性マーカーの読み方

AI が生成した記述に付く根拠の表示。

- 🔵 根拠あり（出典が明記されている）→ そのまま信頼してよい
- 🟡 妥当な推定（根拠は間接的）→ 確認してから使う
- 🔴 根拠なし・要確認 → 必ず確認する

ゲートを通る時点で 🟡・🔴 は残っていてはいけない（確認して 🔵 にするか、対象外として畳む）。スナップショットに 🟡🔴 が残っていたら、それが受入れ前に見るべき注意点。

## よくある質問

**Q. 小さな修正でも全フェーズ通すの?**
A. 通さない。タイプとルートで実施フェーズが絞られる。chore や bugfix は P3・P4 を通らない。

**Q. ADR や設計書は毎回書くの?**
A. 書かない。選択肢のある判断や「守るべき振る舞い」が生まれたときだけ。判断が無ければ書かないのが正しい。

**Q. プロセスが重いと感じたら?**
A. 振り返り（P8）で挙げる。形骸化した記載・不要な承認は playbook の改善対象。プロセス自体を直していく前提で運用する。

**Q. skill を使わず直接 AI に頼んでもいい?**
A. よい。skill はガイド。慣れたら直接指示でかまわない。ただしIssue・受入れ条件・テストの 3 点は、どの進め方でも質を支える芯になる。
